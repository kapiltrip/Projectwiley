I. INTRODUCTION  
A digital integrated circuit is designed and analyzed through a tech-  
nology library. Each cell or macro in a library is characterized for its  
timing, power, and so on, through table entries. Pin-to-pin delay, for  
example, is modeled by a 7-by-7 table indexed by load capacitance  
and input transition time. Missing table entry, upon request from  
timing analysis, is extracted through interpolation with four neigh-  
bor entries as [1]. A popular bilinear interpolation is not accurate  
enough, especially when load capacitance or input transition time is  
large. Our test indicates that cell delay can be inaccurate by 47 ps in  
case of INV in 55-nm technology; the error of path delay can be as  
high as 9%.  
Recently, there have been a few approaches to improving the  
accuracy and runtime of timing analysis using machine learning  
(ML) models. One method uses multilayer perceptron (MLP) and  
autoencoder to predict the output waveform of logic cell from an  
input waveform [2], where the autoencoder encodes the input wave-  
form into a few parameters that become MLP input and decodes  
the output of MLP to the output waveform. This yields errors  
of up to 19% in INV delay even though a signiﬁcant number  
of simulations are used as training samples. In [3], MLPs are  
employed to reﬁne initial timing estimation at the driving point,

II. ML INTERPOLATION  
The timing library includes the timing parameters of individual  
cells, such as cell delay, output transition time, and timing con-  
straints, that describe the timing characteristic of standard cells. They  
are characterized by simulation and stored in a timing table indexed  
by the input conditions. Our problem is to predict unknown tim-  
ing parameters for given input conditions using table entries, where  
the goal is to minimize errors compared to the exact simulation  
results.  
A. Input Features  
There are three types of input features: input conditions of the gate,  
three input maps, and distribution vectors encoded from a timing  
table. Fig. 1 illustrates three input maps, such as timing parameter  
map, row-index map, and column-index map. They have 8×8 pixels  
and their four entries in the center (highlighted in red) are aligned  
to those surrounding input conditions (i, j) in the timing table, where  
i and j are real numbers representing input transition time or load  
capacitance. The pixel of each map is associated with the timing  
parameter, row-index, or column-index at the corresponding position,  
and the pixels outside the table are assigned 0. For each map, the  
values of pixels are scaled between 0 and 1.

IV. CONCLUSION  
We have addressed ML interpolation to get more accurate timing  
parameters. The R-CNN model is employed to allow that the interpo-  
lation can be affected by various ranges of table entries, and VAE is  
used to extract the distribution features of the table. Additionally,  
ML interpolations are parallelized in GPU to decrease the run-  
time overhead of many operations. Experimental results demonstrate  
that ML interpolation reduces timing parameter error by 19.7%  
and path delay error by 3.4% in comparison to bilinear interpo-  
lation at the cost of 13% runtime overhead on average of test  
circuits.  
ML interpolation can be utilized efﬁciently in conjunction with  
bilinear interpolation in such a way that ML interpolation is only  
applied in situations where there are large nonlinearities. It can also  
be applied to other applications, such as the timing library of the  
current source model and the power library including the information  
of internal energy. They are included in our future work.

C. Analysis of Runtime  
The runtime of STA using bilinear interpolation is shown in col-  
umn 9 of Table II. On average of test circuits, bilinear interpolation  
accounts for 4.4% of total STA runtime. Since the number of arith-  
metic operations in MLP of R-CNN is roughly 50 times greater than  
bilinear interpolation, the runtime increases signiﬁcantly; for instance,  
fft_256 takes 2.8 times runtime (2129 s) with ML interpolation. To  
reduce the runtime overhead, ML interpolation is parallelized in GPU,  
and the runtime increase (ꢀTime) is reduced to 13% on average of  
test circuits, as shown in the last column. For reference, parallelizing  
ML interpolation in 16-thread of CPU gives 26% runtime increase.  
Bilinear interpolation has a small number of operations and thus  
does not beneﬁt from parallelization. Rather, the runtime increases  
due to the data transfer time. For example, parallel bilinear inter-  
polation in GPU takes 926 s in fft_256, which is 21% longer than  
before.
